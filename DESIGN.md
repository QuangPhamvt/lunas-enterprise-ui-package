## Brand & Style

Lunas UI is a functional enterprise design system for Lunas Enterprise applications. The visual language is grounded in clarity and restraint — no decorative gradients, no blur effects, no animation for its own sake. Structure is communicated through well-proportioned spacing, a tight typographic hierarchy, and precise color semantics.

The brand color is **Violet** (Violet-500), chosen for its high saturation on light backgrounds while remaining professional and approachable. It avoids the coldness of pure-blue enterprise palettes while staying within the expected register of business software. The overall aesthetic is flat with a deliberate shadow system for depth — components feel crisp and dense, built for data-heavy interfaces like tables, forms, and dashboards.

## Colors

The system organizes color into four brand tones and four semantic states. Every color exposes eight intensity levels under a consistent naming convention: `{color}`, `{color}-foreground`, `{color}-muted`, `{color}-weak`, `{color}-strong`, `{color}-intense`, `{color}-bg-subtle`, and `{color}-border-subtle`.

All source values are authored in **OKLCH**, a perceptually uniform color space. This ensures that stepping across intensity levels produces consistent visual contrast regardless of hue — particularly important for status colors, where bg-subtle backgrounds and intense text must pair reliably.

- **Primary (Violet):** Brand identity color. Used for CTAs, focus rings, links, active navigation states, and input outlines on focus.
- **Secondary (Gray):** Supporting actions, secondary buttons, and low-emphasis chrome.
- **Muted (Zinc):** Table header backgrounds, hover fills, placeholder text, and disabled surfaces.
- **Accent (Slate):** Sidebar selected states and secondary highlights.
- **Warning** is the only status color with a dark foreground (`warning-foreground: #78350F`) — amber is too bright for white text at its base value.
- **Text Positive/Negative:** Separate token families for light surfaces (`text-positive`) and dark or colored surfaces (`text-negative`), avoiding opacity hacks that would muddy against colored backgrounds.

## Typography

Lunas UI uses **Inter** exclusively — a neutral, geometric sans-serif that holds up at the small sizes and high densities common in enterprise UIs. No display or serif faces are used.

The effective type scale has two origins:
1. **Editor headings** (H1–H3) from `styles/text-editor.css` define heading sizes for ProseMirror-based rich text editors.
2. **UI type** (`body-md`, `label-sm`, etc.) comes from the Tailwind `text-*` utilities used in component variants — `text-xs` (12px) through `text-sm` (14px) cover the majority of interface text.

Font weight steps up one tier at small label sizes (from 400 to 500) to maintain legibility without a size increase. A global letter-spacing of `-0.015625rem` (≈ −0.25px) tightens the Inter stack slightly, giving it a more considered, professional feel at UI sizes.

## Layout & Spacing

All dimensions are multiples of a **4px base unit**. The practical spacing atoms:

- `4px` — tight icon gaps, inline badge padding
- `8px` — standard gap between grouped elements
- `12px` — comfortable component internal padding
- `16px` — card padding, section-level gaps
- `24px` — container padding, dialog padding
- `40px` — major layout section separation

Table cells use compact `10px 16px` padding to support high-density data views. Cards and dialogs use `24px` internally for readability at full-page scale.

## Elevation & Depth

Depth is expressed through a six-level shadow scale. Each level uses a compound approach: a `0 0 0 1px` ring provides edge definition against the near-white page background, while additional blur layers add perceived lift.

- **btn** — Minimal lift. An inset top highlight simulates a slightly physical button surface.
- **input** — Barely visible. Intentionally subtler than buttons so form fields remain visually passive.
- **nav** — Three-layer shadow anchors the navigation bar to the top of the viewport without a heavy border.
- **card** — Moderate lift to separate card surfaces from the page.
- **dropdown** — Strong enough to clearly break from the content plane, critical for popover usability.
- **dialog** — Maximum. A 48px spread communicates the topmost modal layer.

No component uses `backdrop-filter` or blur. All shadows operate against the `#FAFAFA` page background.

## Shapes

Shape language uses **TailwindCSS v4 `rounded-*` utilities**:

- `rounded-sm` (4px) — Form controls: inputs, buttons, checkboxes, select triggers. The tightest radius, communicating precision and density.
- `rounded-md` (8px) — Text editors and medium containers.
- `rounded-lg` (12px) — Cards, alerts, dialogs, dropdowns. The primary container radius.
- `rounded-xl` (16px) — Large modal sheets and bottom drawers.
- `rounded-full` — Badges, pills, and avatars.

Buttons and inputs deliberately use `rounded-sm` rather than `rounded-full`. The system avoids the pill aesthetic of consumer products in favor of a crisper, utilitarian feel suited to dense enterprise layouts.

## Components

### Button

The button system covers **6 variants × 8 colors × 7 sizes**. Variants express emphasis level; colors express semantic intent; sizes adapt to context (dense table row actions use `xs`, primary page CTAs use `lg` or `xl`).

- `default` — Solid fill. Reserved for primary actions only.
- `outline` — Bordered, high-contrast. For secondary actions that must compete visually with default buttons nearby.
- `soft` — Tinted background. A gentler secondary option when a border feels too strong.
- `subtle` — Tinted background with an outline. Combines presence with softness; used for confirmable secondary actions.
- `ghost` — Transparent. Contextual actions inside tables, toolbars, and icon button groups.
- `link` — Text only with underline. Inline anchors within copy, never standalone UI actions.

### Input

Five visual variants let inputs adapt to context: `outline` is the default for standard form fields; `ghost` is used for embedded search or in-table editing; `soft` and `subtle` for lightly styled sidebar inputs. All variants share one error state — `aria-invalid` activates `danger-bg-subtle` background, `danger` outline, and `danger-weak` ring, consistent regardless of variant.

### Alert

All five alert variants (`default`, `destructive`, `warning`, `success`, `info`) use the same structural pattern: `{color}-bg-subtle` background + `{color}-intense` text + `{color}-border-subtle` border. The most muted tint as background paired with the most intense shade as text always produces sufficient contrast without using full-saturation fills that would compete with the page content.

### Table

Table headers use `muted-bg-subtle` and `text-positive-weak` to recede visually behind the data rows. Row states use the same muted color family stepped by intensity: hover is `muted-bg-subtle`, selected is `muted-muted` (one step darker). Pinned columns receive a matching `border-border` hairline to maintain visible separation between fixed and scrolling regions.

### Sidebar

The sidebar uses `zinc-100` (`--sidebar: #F4F4F5`), one step darker than the page background `zinc-50` (`#FAFAFA`). This passive tonal distinction separates navigation from content without requiring a high-contrast panel. Active navigation items use the full primary Violet; hover states use `muted-weak` fill.

### Text Editor

Two variants wrap ProseMirror-based editor surfaces: `outline` applies a border and background for use as a form field, `ghost` removes the border for full-bleed editing surfaces. Both inherit editor content styles from `styles/text-editor.css` — heading scales, list spacing, code block padding, and blockquote treatments are all defined there.

### Dialog & Dropdown

Dialogs use the `shadow-dialog` level (48px spread) and `rounded-lg` with 24px internal padding. Dropdowns use `shadow-dropdown` and `rounded-md` with 4px internal padding for tight item lists. Both rely on a `0 0 0 1px` ring as their primary edge definition rather than a visible border.
